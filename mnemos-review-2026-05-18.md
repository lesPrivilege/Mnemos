# Mnemos：一个结构化学习工具的架构复盘

Mnemos 是一个间隔重复记忆 + 题库练习 + 阅读的移动端学习工具，三天用 vibe coding 工作流独立完成。这篇文章复盘它的核心设计决策——为什么应用层不集成 LLM API，以及这个选择如何在三个模块中各自兑现。

## 项目背景

Mnemos 的起点不是产品规划，是三个独立需求的自然聚合。

工具链的变化提供了一个契机。此前的工作流以 Claude Code TUI 和其他 frontier 客户端为主，主要用于科研伴读和学术文本处理，coding 需求量不大。入选 MiMo 百亿 token Plan 后，大规模推理成本的约束被解除，开始尝试 OpenCode + MiMo 的 vibe coding 工作流——让模型做体力执行，人做架构判断。

三个模块分别来自三个具体需求：

**闪卡模块**是最直接的。备考过程中一直在用 Anki，但它有两个问题：UI 过于古朴，功能层级复杂到影响了日常使用意愿。对于需要长期坚持的间隔重复实践，工具的使用摩擦直接决定坚持率。做一个干净、移动端可用的 SRS 工具成了明确的待办项。

**题库模块**解决另一个瓶颈。备考的科目以自命题为主，教辅资料没有现成的结构化题库。传统的做法是买习题集、手写答案——既慢又无法利用间隔重复的调度优势。本地已有 OCR 清洗和结构化提取的 skill，把这条管线接到 Mnemos 上，就能把 PDF 教辅转化为可练习的 questions.json。LLM 在管线中的角色不是自由生成，而是在 schema 约束下做语义修补和格式规范化。

**阅读模块**的需求来自另一个方向。CLI 环境中长期使用 reading-companion skill 处理 arxiv 论文、GitHub repo、frontier 文章的结构化摘要。这些摘要积累下来需要一个移动端的消费容器——在手机上舒适地阅读，支持 LaTeX 渲染、高亮标注和书签。

三个模块从三个方向独立生长出来，彼此之间没有强耦合。这种解耦不是架构设计的刻意追求，而是需求的自然属性——记忆、练习、阅读是三件不同的事，它们的唯一交集是同一个用户在做。保持模块边界清晰，意味着每个模块可以独立迭代而不影响其他两个。

## 核心设计判断：应用层只消费结构化输出

Mnemos 在架构上做了一个边界划分：程序内不调用任何 LLM API。所有需要模型参与的环节——闪卡生成、题目清洗、文档整理——都在外部完成，Mnemos 只接收结构化的 markdown、JSON、纯文本。

这个决策的初始动机很务实：在 React + Capacitor 应用里集成 LLM API 麻烦，外部化省事。但跑通之后发现，这个边界恰好让应用层获得了对模型层变化的抗折旧能力。应用逻辑稳定在格式解析、调度算法、渲染管线上，不随 prompt 迭代或模型升级而需要修改。内容生成的复杂度——context 组装、prompt 工程、输出校验——被隔离在外部 CLI 工具和 skill 中。

下面分别看三个模块如何在这个边界上运作。

## 闪卡模块：格式消费者，不是生成者

闪卡模块的工作流：用户拿教辅资料的 chunk 喂给 chatbot，chatbot 按 prompt 模板生成结构化 markdown，用户将 markdown 导入 Mnemos。

`formatSpec.js` 是这个模块的单一事实来源。它同时定义了 prompt 模板和解析规则。prompt 模板精确规定了 markdown 的结构——`#` 科目、`##` 章节、`-` 知识点标题、缩进行释义——以及内容约束：不添加原文没有的内容、不添加编号和 emoji、公式用 LaTeX。`mdParser.js` 的解析逻辑与 prompt 模板严格对应：逐行扫描，标题推进层级上下文，bullet 行触发 flush 生成卡片，缩进行累积为卡片背面。

控制权的分布值得注意：用户决定用什么材料、切多大 chunk、用哪个模型。Mnemos 不参与内容生成决策，只执行格式契约——你按我的格式给，我按规则解析。解析器的容错策略也很明确：不符合格式的行被跳过，不崩溃、不猜测意图。

SM-2 调度是应用层的核心逻辑。`sm2.js` 实现标准 SM-2 算法，quality 映射到四个评分按钮（Again=1, Hard=2, Good=4, Easy=5）。`scheduler.js` 管理到期判定和排序，按 dueDate 升序拉取到期卡片，支持每日上限。调度逻辑完全独立于内容来源——无论卡片是用户手写还是 LLM 生成，复习节奏由算法决定，不受内容生产侧变化的影响。

复习会话的持久化也体现了务实的设计判断。`reviewSession.js` 只保存 deck identity 和计数，不保存完整队列——因为 SM-2 分数已经持久化，恢复时重新拉取即可。24 小时过期，过期会话自动清除。这个设计避免了序列化大队列的复杂性，也避免了过期状态误导用户。

## 题库模块：LLM 是被管线管理的执行器

题库的数据流比闪卡复杂得多。原始教辅资料经过 OCR 产生充满噪声的文本，需要经过结构化处理后变成标准化的 `questions.json`。这个处理不发生在 Mnemos 内部——本地 CLI 做文档清洗，LLM 做语义修补，最终产出符合 JSON schema 的题目文件。

`questionParser.js` 是这个模块的守门人。它的验证逻辑是逐层防御的：id 缺失则跳过，type 不在允许集合（choice/review）则跳过，选择题缺少选项则警告。向后兼容策略也清楚：旧的 calculation/coding/essay 类型自动映射到 review。标准化流程涵盖 source 推断、chapter 格式统一（`2 数据的表示` → `第2章 数据的表示`）、section 从 id 解析、选项尾部 ` - ` 清理。

这种管线式约束和 Claude Code 的 tool use 设计是同构的。LLM 不是自由行动的 agent，而是被管线管理的执行器——它的输出必须符合预定义的 schema，不符合的被拒绝或修补。管线定义了什么算有效，LLM 在管线划定的边界内工作。这个设计让题库的数据完整性不依赖 LLM 的一次性输出质量，管线可以在发现错误时重新调用模型修补特定条目，而不影响已通过校验的题目。

`quizEngine.js` 进一步体现了应用层与内容层的分离。练习模式（sequential/random/wrong/new/due/starred）都是纯数据操作——过滤、排序、切片。错题本的判定逻辑也完全基于本地 progress 数据：曾经做错且未连续做对 2 次以上。没有任何判断依赖 LLM。

## 阅读模块：纯 context 消费

阅读模块在三个模块中边界最清晰：不做生成、不做分析、不调用 LLM。它的全部功能围绕一件事——在移动端舒适地阅读上游已完成的结构化 markdown。

`renderDoc.js` 是阅读模块的渲染入口。管道很简单：md 直接渲染，tex 先经过 `texToMarkdown()` 转换为 markdown 再渲染，txt 做 HTML 转义后包在 pre 标签里。`texToMarkdown()` 是一个务实的简化转换器——处理章节标题、数学环境、定理环境、列表、行内格式，删除注释和 label/ref，不做完整的 TeX 语法树解析。它针对的是「阅读数学教材」这个具体场景，不是通用 TeX 渲染器。

`renderMarkdown.js` 是整个项目共享的渲染基础设施。marked 做 markdown → HTML，KaTeX 做 LaTeX 渲染（lazy import，只在检测到 `$` 字符时才加载），DOMPurify 做 XSS 防护。渲染器同时支持 inline math（`$...$` 和 `` `$...$` ``）、display math（`$$...$$` 和 code block with `$$`）、以及 marked 自定义扩展的 mathBlock/mathInline tokenizer。sanitize 的白名单精确到 tag 和 attribute 级别。

阅读器的 UI 决策有几个值得注意的点。进入时 chrome（顶栏和底栏）默认隐藏，点按内容区切换显示——沉浸阅读优先。顶栏只保留返回和标题，功能按钮全部移至底部栏（TOC、高亮、书签、设置）。滑动不再触发 bar 显示，避免打断阅读。设置展开在底部栏上方，即时调整字号（14-24px）、行距（1.4-2.2）、边距（12-40px），默认值 18/1.8/24。iPad 适配用 max-width 680px 限制行长，不依赖媒体查询。

高亮和书签的设计也保持了 local-first：高亮从 DOM textContent 提取上下文 snippet（前后各 60 字符），保存到 localStorage；书签保存滚动百分比，恢复时用 scrollTo 定位。导出高亮按 TOC 标题分组输出 markdown。

## 与 wquguru Harness Engineering 框架的对应

事后回看，三层恰好对应 wquguru 框架的三个核心概念：

- 闪卡 = **Prompt**：用户持有控制权，选材料、切 chunk、换模型。应用层是格式消费者，不参与内容生成。
- 题库 = **Harness**：结构化管线约束 LLM 输出。schema 定义合法性边界，不符合的产出被管线拒绝，LLM 是被管理的执行器。
- 阅读 = **Context**：消费上游结构化产出。渲染、导航、标注，不做进一步变换或生成。

需要诚实标注的是：这个对应是事后发现的，不是按框架设计的。Mnemos 开发在前，读 wquguru 在后。两个独立工作者在不同语境下得出相同结论——文献学里叫「暗合」。暗合的权重比一个引用另一个更高，因为它说明这个结论不是追随某个框架的结果，而是从各自的工程约束中独立推导出来的。

## 工具链与协作模式

实际使用的工具链：OpenCode + MiMo 做主力执行，Claude 做判断层。分工很明确——MiMo 负责体力活（组件搭建、样式调整、文件操作），Claude 负责架构决策（模块边界划分、数据流设计、API 设计）和 code review。这个分工在实际工作中运转良好：MiMo 的执行速度（不需要来回沟通）和 Claude 的判断质量（上下文窗口大、推理深度够）各自发挥了长处。

vibe coding 的体验是：三天从零到可用的 prototype，但这个速度需要两个前提——对目标场景有清晰的心智模型（自己就是用间隔重复的用户），以及工具链的熟练度（知道什么时候让 MiMo 冲，什么时候停下来让 Claude 判断）。

## 边界之外

这个架构选择当然有代价。外部化意味着用户需要维护一套独立的工具链来生成内容。导入流程增加了操作步骤。但这些代价在当前阶段是可接受的，因为内容生成本身就是需要用户深度参与的过程——选择材料、判断质量、调整 prompt——不应该被自动化掉。把内容生成外部化，不是逃避工程复杂度，而是承认这个环节的控制权属于用户。
