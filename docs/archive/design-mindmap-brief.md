# Design Brief — 导图视图（决策稿，非实现文档）

目的：在实现前把功能逻辑定下来。看完后对「待决策」四条给出选择，我再出
implementation prompt。

## 定位（建议，可推翻）

**现有层级数据的只读可视化 + 掌握度着色 + 导航入口。不做可编辑思维导图。**

依据：三个模块的数据天然是树——

| 模块 | 层级 | 叶子 | 掌握度信号 |
|---|---|---|---|
| 闪卡 | deck → chapter → section | card | easiness / repetitions / lapses |
| 题库 | subject → chapter → section | question | progress status / streak |
| 阅读 | doc → TOC 标题树 | 段落 | 无（只有进度%） |

导图不需要新数据结构、不需要持久化任何东西，是纯派生视图。「可编辑」才是重功能
（节点 CRUD、布局持久化、与源数据双向同步），砍掉编辑，成本降一个数量级。

## 分阶段

**P0 — cardStats.js（独立价值，先行）**
`src/lib/cardStats.js`：`mastery(card) → 0..1`，EF/repetitions/lapses 加权（旧
iteration prompt 提过的模型）。聚合版 `masteryOfCards(cards)`。不只导图用——
Browse 列表、DeckDetail 章节行都可以加掌握度指示。单独一个分支，先落。

**P1 — 树形大纲视图（导图的 80% 价值、20% 成本形态）**
DeckDetail / SetDetail 加视图切换（列表 ⇄ 结构）。结构视图 = 可折叠缩进树：
chapter → section 节点，每节点右侧掌握度条（P0 的聚合值着色），点击叶子层级进入
对应章节复习/练习。纯 DOM，无缩放平移，移动端交互零风险。

**P2 — 图形化导图（真·导图形态，P1 验证需求后再决定做不做）**
放射状/横向树布局 + 缩放平移。这一步才有渲染选型问题。

## 待决策

1. **P1 的树形大纲是否足够？** 我的判断：在手机竖屏上，可折叠树的信息密度和可用
   性优于强行塞进小屏的图形导图；P2 可能永远不需要做。
2. **掌握度着色方案**：连续色阶（红→赭→绿）还是三档（弱/中/稳）？三档更可读。
3. **阅读模块要不要进这个体系？** 阅读没有掌握度信号，TOC 树已有（Reader 面板），
   建议阅读不做，导图只覆盖闪卡 + 题库。
4. **P2 若做，渲染选型**：markmap（现成、但 markdown 输入需要转换层、样式难融入）
   vs 自绘 SVG 树（d3-hierarchy 布局算法 + 手写渲染，样式完全可控，缩放平移用
   pointer events 自己写，成本主要在这）。倾向后者，但仅在 P1 被证明不够时。

## 非目标（无论哪阶段）

节点编辑/新增/拖拽；导图数据持久化；跨模块混合视图（一张图同时画闪卡+题库）；
导出图片。
